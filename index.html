20行でわかる！OpenMPWとOpenLANEとSkywaterPDKの様々

OpenMPWの概要

  • 無料でLSIを焼いてくれるプロジェクト
  • 現在MPW-ONEとMPW-TWOが完了していて現在MPW-THREEの募集中
  • Caravel(RISC-V+IOのフレーム)のUser Project Areaに自分のデザインを実装する
  • IBMとかもデザインを投稿している

必要要件

  • ライセンス諸々
  • 再現可能なGDSIIレイアウト
  • Caravelのテストハーネスとパッドフレームの使用
  • OpenLANEフローのOpenMPWプリチェックツールでの合格

まとめ

  • Caravelをforkして改造
  • プリチェックツールで通るならおｋ

各種リンク

  • OpenLANE : RTLからGDSII吐くやつ
  • Caravel : パッドフレーム
  • Caravel User Project
  • プリチェックツール
  • MPW-ONE
  • MPW-TWO

LSIを焼こう！[1] (未完成)

概要はefablessの動画[2]を見れば大体わかる。また使うツールのバージョンやCaravelの概要はこのドキュメント[3]に書いてある。シャトル毎に形式が変わるのでMPW3の概要が出たら更新する。

9/17にMPW3がOpenになりました、早急に対応します。

Analogも対応出来るみたいだけど需要があったら書く

必要なもの

  • Caravel_user_project : これのUser Project Areaに自分のデザインを加える
  • PDK
  • OpenLANE : RTLをGDSII[4]に変換するツールフロー
  • 自分のデザイン : Wishboneと接続出来るやつ
  • 時間 : いっぱい

1.各種インストール & ビルド

1.Caravelのインストール

基本的に、forkしたCaravel_user_projectのリポジトリ[5]上で全てを進めていく




git clone git@github.com:efabless/caravel_user_project.git -b mpw-3
cd caravel_user_project
make install

2.PDKのビルド

5分くらい掛かる




mkdir ~/pdks
export PDK_ROOT=~/pdks
make pdk

3.OpenLANEのインストール

Dockerは事前にインストールしてデーモンを起動しておく。




mkdir ~/openlane
export OPENLANE_ROOT=~/openlane
make openlane

2.デザインを自作する

Caravelの概要

我々はcaravelのUser Project Areaに自分のデザインを挿入する訳だが、そのデザインを作り始める前に、User Project Areaについて多少知っておく必要がある。
以下がCaravelパッドフレームの概略図

User Project Areaについて抑えておくべき事項は3つ

  • Wishboneバスが繋がっている(32bit)
  • Logic Analyzer Signalsが繋がっている(128bit)
  • GPIOが生えている(38本)

Logic Analyzer Signalsは図におけるDebug Lineと同じである。なぜ図ではDebug Lineになってるかといえば、それはそう図を作っちゃったからである。作っちゃったもんは仕方ない。

Wishbone、Logic Analyzer Signalsに関してはCPU側からMMIOを通してアクセスが可能であり、Wishboneのアドレスは0x30000000から0x80000000-1の間で自由に設定できる。

以下では、Wishbone、Logic Analyzer Signals、GPIOそれぞれの扱い方を説明する。

Wishboneを扱う

という訳でWishboneの扱い方を付け焼き刃程度に解説する。既にWishboneについて知ってる方はこの部分を読む必要は無い。

Caravelでは10種類のWishboneに関係した線が存在している。

  • wb_clk_i
    マスターから入力されるクロック。この立ち上がりのタイミングで値が読まれる。
  • wb_rst_i
    リセット信号。wishboneインターフェイスのステートマシンをリセットさせる。IPコアをリセットする必要は無い。
  • wbs_stb_i
    ストローブ入力。スレーブが選択されている事を示す。
  • wbs_cyc_i
    サイクル入力。有効なバスサイクルが進行している事を示す。
  • wbs_we_i
    ライトイネーブル入力。現在のバスサイクルがReadかWriteか示す。1でWrite、0でRead。
  • wbs_sel_i[3:0]
    セレクタ入力。入力データ(wbs_dat_i[31:0])の内、有効なデータがどれか示す。8ビット粒度。
  • wbs_dat_i[31:0]
    データ入力
  • wbs_adr_i[31:0]
    アドレス入力
  • wbs_ack_o
    アクノリッジ出力。1にすると正常なバスサイクルの終了を示す。
  • wbs_dat_o[31:0]
    データ出力

とまあ沢山信号線があって制御がダルいように思うかもしれないが、割とシンプルな回路で扱える。

まず信号を整理する。[6]












// WB MI A
assign valid = wbs_cyc_i & wbs_stb_i; 
assign wb_ack_o = ready;
assign we = wbs_we_i;
assign wbs_dat_o = rdata;
assign wdata = wbs_dat_i;
assign addr = wbs_adr_i;
// wb_sel_iはお好みで

assign reset = wb_rst_i;
assign clk   = wb_clk_i;

そしてシンプルなデータ読み出し、書き込みが出来るロジックを以下に書いた。































always @(posedge clk) begin
    if(reset) begin
        // ここにリセットの処理
        ready <= 1'b0;
    end else begin
        if(ready) begin
            ready <= 1'b0;
        end
        // Read
        if (valid && !ready && !we) begin
            case(addr)
                // こんな感じでRead処理
                ADDR1: rdata <= nanka_data1;
                ADDR2: rdata <= nanka_data2;
                // ・・・
            endcase
            ready <= 1'b1;
            
        // Write
        end else if (valid && !ready && we) begin
            case(addr)
                // こんな感じでWrite処理
                ADDR8: nanka_reg1 <= wdata;
                ADDR9: nanka_reg2 <= wdata;
                // ・・・
            endcase
            ready <= 1'b1;
        end
    end
end

あとはこれをたたき台にして、ここのアドレスに書き込んだらスタート〜とかこのアドレスのレジスタがHighになったら処理完了〜とか適当な回路を作れば大体なんとかなる。

Logic Analyzer Signalsを扱う

ついでにLogic Analyzer Signalsの扱い方も解説する。

Caravelでは3種類のLogic Analyzer Signalsに関連した信号線が存在している。

  • la_data_in[127:0]
    入力データ、128ビット。
  • la_data_out[127:0]
    出力データ、128ビット。
  • la_oenb[127:0]
    出力イネーブル。Highで出力、Lowで入力。これはSoC側がMMIOで1ビットずつ設定するので読むことしか出来ない。

まあ流れで扱える

assign data1 = la_data[31:0];

とか




always @(posedge clk) begin
    la_data_out[63:32] <= data2;
end

とか

GPIOを扱う

GPIOの扱い方を解説する。

Caravelでは3種類のGPIOに関連した信号線が存在している。

  • io_in[37:0]
    GPIO入力
  • io_out[37:0]
    GPIO出力
  • io_oenb[37:0]
    GPIO出力イネーブル。Highで出力、Lowで入力。

ピンの位置はここ[7]のmprj_io。

割り込み

次に割り込みについて説明する。CaravelではUser Project Areaから割り込み信号を出す事が可能である。

割り込みは、信号線user_irq[2:0]の内どのビットを立てても引き起こす事が(多分)可能である。 割り込みベクタはアドレス0である。正確な情報はこの動画[8]以外に存在しない。

またサンプルコード[9]を一度目を通すことを強くおすすめする。

以上を組み合わせてキミだけのデザインを自作しよう！！

3.デザインをCaravelに載せる[10][11]

基本的に以下の流れで作る

  1. caravel_user_project/verilog/rtl/user_project_wrapper/に自分のデザインを入れる
  2. config.tclを編集する

1.user_project_wrapperに自分のデザインを書き込む

caravel_user_project/verilog/rtl/user_project_wrapper.vのIOをそのままに中身を自作のデザインに置き換えると、ピン割り当ての設定ファイルをそのまま利用できるので非常に楽。[12]
OpenMPWでは./verilog/rtl以下に自分のデザイン用のディレクトリを作るのを推奨している。[13]

SoCとのデータ転送には基本的にWishboneバスを利用するので、Wishboneの仕様を各自で調べ、仕様に沿ったIOを持ったデザインを自分で作る必要がある。

GPIOに関してはUser Project Areaからは謎のテクノロジーでinput/output/inenb/outenbに分離されている[14]ように見えるので深く考えずに自分のデザインと繋げてよい。

Wishboneと接続するに当たり、ユーザーは自作のデザインにアドレスを0x30000000から0x80000000-1までで自由に割り当てできる。[15]

2. config.tclを編集する

user_project_wrapper.vのみを編集する場合は特に何もしなくてよいが、自作のデザインは複数のVerilogファイルから構成される場合が殆どなので、GDSIIに変換する前にVerilogファイル達をOpenLANEに教えてやる必要がある。そのためにconfig.tclを編集する。[13:1]

まずは環境変数を２つ追加する。以下をビルドしたPDKとopenlaneのパスに各自書き換える。



export PDK_ROOT=/path/to/pdk
export OPENLANE_ROOT=/path/to/openlane

次に、caravel_user_project/openlane/user_project_wrapper/config.tclの以下の変数を編集する。










set ::env(VERILOG_FILES_BLACKBOX) " \
      $script_dir/../../caravel/verilog/rtl/defines.v \
      $script_dir/../../verilog/rtl/user_proj_example.v"

set ::env(EXTRA_LEFS) " \
   $script_dir/../../lef/user_proj_example.lef"

set ::env(EXTRA_GDS_FILES) " \
   $script_dir/../../gds/user_proj_example.gds"
  • VERILOG_FILES_BLACKBOXにはuser_project_example.vを消して自分のVerilogファイル群を書き込む。
  • defines.vはそのままでよい。
  • EXTRA_LEFSとEXTRA_GDS_FILESにはuser_proj_example.(lef/gds)を消して、代わりにモジュールuser_project_wrapperの一階層下のモジュールの名前を入れる。[16]

最後にMACRO_PLACEMENT_CFGの行を削除する。

set ::env(MACRO_PLACEMENT_CFG) $script_dir/macro.cfg

3.テストベンチ/ファームウェアを書く

オレオレLSIが完成したら、それが正しく動作するか確かめる必要がある。そういうわけでテストベンチ又はファームウェアの書き方を解説する。サンプルはverilog/dv/以下に色々入っているので参考にするとよい。

verilog/dv/以下で作業を行う。

General

ここではCaravelでファームウェア又はテストベンチの実行方法を解説する。

コンパイルとシミュレーション環境に関してはDockerイメージが提供されている。

docker pull efabless/dv_setup:latest

環境変数の設定




export PDK_PATH=<pdk-location/sky130A>
export CARAVEL_ROOT=<caravel_root>
export UPRJ_ROOT=<user_project_root>

コンテナをスタート

docker run -it -v $CARAVEL_ROOT:$CARAVEL_ROOT -v $PDK_PATH:$PDK_PATH -v $UPRJ_ROOT:$UPRJ_ROOT -v $PDK_ROOT:$PDK_ROOT -e CARAVEL_ROOT=$CARAVEL_ROOT -e PDK_PATH=$PDK_PATH -e UPRJ_ROOT=$UPRJ_ROOT -e PDK_ROOT=$PDK_ROOT -u $(id -u $USER):$(id -g $USER) efabless/dv_setup:latest

dvディレクトリに移動

cd $UPRJ_ROOT/verilog/dv/

テストが入ったディレクトリでmakeしてシミュレーションを走らせる。



cd <dv-test>
make

SIM=GL makeでゲートレベル[17]のシミュレーションも出来る。



cd <dv-test>
SIM=GL make

ここまで解説してもそもそもプログラムとかテストベンチとかMakefileの書き方とか何もわからないと思うので、以降ではそれぞれ解説していく。

Makefileの書き方

他のdv以下のディレクトリからコピペでいい。ただ43行目あたりにPATTERN = 〜と書いてある行があるので好きな名前に変えるとよい。

User Project Areaから外部へ出ているGPIO

Configuration

このGPIOは38本存在するが、それぞれにコンフィグ用のレジスタ(グローバル変数として扱える)reg_mprj_io_0~reg_mprj_io_37が存在する。それぞれに以下の値を設定出来る。

値 説明
GPIO_MODE_MGMT_STD_INPUT_NOPULL 自作デザインへの入力に設定(謎)
GPIO_MODE_MGMT_STD_INPUT_PULLDOWN 自作デザインへの入力に設定(プルダウン)
GPIO_MODE_MGMT_STD_INPUT_PULLUP 自作デザインへの入力に設定(プルアップ)
GPIO_MODE_MGMT_STD_OUTPUT 自作デザインからの出力に設定
GPIO_MODE_MGMT_STD_BIDIRECTIONAL 自作デザインの入出力に設定
GPIO_MODE_MGMT_STD_ANALOG 謎
GPIO_MODE_USER_STD_INPUT_NOPULL SoC側への入力に設定(謎)
GPIO_MODE_USER_STD_INPUT_PULLDOWN SoC側への入力に設定(プルダウン)
GPIO_MODE_USER_STD_INPUT_PULLUP SoC側への入力に設定(プルアップ)
GPIO_MODE_USER_STD_OUTPUT SoC側からの出力に設定
GPIO_MODE_USER_STD_BIDIRECTIONAL SoC側との入出力に設定
GPIO_MODE_USER_STD_OUT_MONITORED 謎
GPIO_MODE_USER_STD_ANALOG 謎
Activation

GPIOを有効にするためのレジスタ(グローバル変数として扱える)reg_mprj_xferが存在する。これをHighにすれば上記の設定内容が適用され、GPIOが有効になるが、その際に以下の記述が必要である[18]。



 reg_mprj_xfer = 1;
 while (reg_mprj_xfer == 1);

一見無限ループだが、謎のテクノロジーでループを抜けるので機械的にコピペすればいい。

Data

GPIOをGPIO_USER_MODE_STD_*に設定した場合にのみ(多分)(未検証)、以下のレジスタを介してGPIOからSoC側にデータを入出力することが可能である。

  • reg_mprj_datal : 最初の32本の値を入出力
  • reg_mprj_datah : 残りの6本の値を入出力

User Project AreaとSoCをつなぐLogic Analyzer線

User Project AreaとSoCは128本のLogic Analyzer線(以下LA線)で接続されており、名前とは裏腹にデータのやり取りが可能である。

Configuration

各LA線の入出力はreg_la0_oenb ~ reg_la3_oenbとreg_la0_ienb ~ reg_la3_oenbという各32ビットのレジスタで設定できる。1でSoC側への入力、0でSoC側から出力と考えればよい[19]。







//  LA probes [31:0], [127:64] をCPUへの入力に設定 
// Configure LA probes [63:32] をCPUからの出力に設定
reg_la0_oenb = reg_la0_iena = 0xFFFFFFFF;    // [31:0]
reg_la1_oenb = reg_la1_iena = 0x00000000;    // [63:32]
reg_la2_oenb = reg_la2_iena = 0xFFFFFFFF;    // [95:64]
reg_la3_oenb = reg_la3_iena = 0xFFFFFFFF;    // [127:96]

一括だけでなく、各ビット毎に設定可能

Data

LA線を介したデータのやり取りはreg_la0_data ~ reg_la3_dataというレジスタを介して行うことが出来る。以下の例では、上記の設定でLA線の[63:32]を出力に設定したため、SoC側からUser Project Areaへ値0xdeadbeefを出力している。





buffer0 = reg_la0_data;    // [31:0]
reg_la1_data = 0xdeadbeef; // [63:32]
buffer2 = reg_la2_data;    // [95:64]
buffer3 = reg_la3_data;    // [127:96]

このLA線を使ってSoC側からUser Project Areaにいい感じに分周したクロックとかリセットを入力出来たりするらしい。需要があったら書く(つかれた)。

ここ[20]らへんを読めば出来る
